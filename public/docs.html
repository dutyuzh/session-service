Overview

Mission: Design and implement a Session Service for a web platform deployed on on-prem infrastructure.

This project demonstrates identity and session lifecycle systems that support both anonymous and authenticated users while meeting scalability, security, and performance requirements.

Problem Statement

Modern web platforms need to manage user sessions across multiple scenarios:

  • Anonymous browsing → Shopping carts, preferences before login
  • Seamless login → Preserve cart when user authenticates
  • Multi-device access → User logs in from phone, laptop, tablet
  • Persistent login → "Remember me" for 90 days
  • Security → Protection against hijacking, fixation, XSS
  • Microservices → Share session data across email, billing, analytics services

⚠️ Rate Limiting Requirements (Production CRITICAL)

The rate limit column above shows recommended limits for production. Current prototype has NO rate limiting implemented.

Color coding:

  • Red background: CRITICAL - Must implement (activity ping is primary DoS vector)
  • Orange background: HIGH - Important for security (brute force protection)
  • Yellow background: MEDIUM - Prevents abuse of expensive operations
  • No color: Rate limiting optional (low abuse risk)

Why this matters: A single attacker can send 10,000 requests/second to POST /sessions/me/activity, saturating 20% of Redis capacity. Without rate limiting, 5 attackers can take down the entire service. See Scalability section for detailed abuse scenarios and implementation guide.

⚠️ Current Implementation: JSON Blobs (Prototype)

Sessions are stored as JSON strings using redis.set() and redis.get(). This is simple and works well for prototypes, but has performance limitations at scale.

Problem: Every activity ping rewrites the entire 5KB session to update a single 24-byte field (lastSeenAt).

  • Write amplification: 5000 bytes written to change 24 bytes (200× inefficiency)
  • Network cost: At 10k pings/sec = 50MB/sec bandwidth
  • CPU cost: JSON.parse() + JSON.stringify() on every operation

For production at scale (>10k req/sec), migrate to Redis Hashes for partial updates. See "Data Model Optimizations" section below.

⚙️ Data Model Optimizations (Production Upgrades)

Performance Issue: Current prototype uses JSON blobs. At scale (10k+ req/sec), this causes write amplification, network waste, and CPU overhead. Migrate to Redis Hashes for production.

Optimization 1: Redis Hashes for Partial Updates

Problem:

  • Every POST /sessions/me/activity rewrites 5KB to update 24 bytes
  • 10,000 activity pings/sec = 50MB/sec write bandwidth (inefficient)
  • JSON parse/stringify adds CPU overhead

Solution: Redis Hashes

// Current: JSON blob (5KB write per ping)
await redis.set(`session:${sid}`, JSON.stringify(session), 'EX', 1800);

// Optimized: Redis Hash (24 bytes per ping)
await redis.hset(`session:${sid}`, 'lastSeenAt', new Date().toISOString());
await redis.expire(`session:${sid}`, 1800);

// Full session structure as hash:
HSET session:abc userId "alice"
HSET session:abc state "authenticated" 
HSET session:abc deviceId "device-xyz"
HSET session:abc lastSeenAt "2026-02-25T10:15:00Z"
HSET session:abc createdAt "2026-02-25T10:00:00Z"
HSET session:abc attributes '{"cart":["laptop"]}' // Still JSON for nested objects

// Events stored separately (capped list):
RPUSH session:abc:events '{"type":"CREATED","at":"2026-02-25T10:00:00Z"}'
LTRIM session:abc:events -50 -1  // Keep only last 50 events

Benefits:

Metric JSON Blob Redis Hash Improvement
Bytes per activity ping 5,000 bytes 24 bytes 200× reduction
Network at 10k req/sec 50 MB/sec 240 KB/sec 200× reduction
CPU overhead JSON parse + stringify Direct field access ~5× faster
Complexity Simple (1 command) Moderate (HSET + EXPIRE) Trade-off

Optimization 2: Cap Lineage/Events Length

Problem:

  • events array grows unbounded (every session rotation, merge, etc.)
  • After 100 operations, session size could be 10KB+ (mostly old events)
  • Older events have diminishing audit value

Solution: Rolling Window

// Current: Unbounded growth
session.events.push(newEvent); // Grows forever

// Optimized: Cap at 50 events (circular buffer)
session.events.push(newEvent);
if (session.events.length > 50) {
  session.events.shift(); // Remove oldest
}

// Or with Redis List (better):
await redis.rpush(`session:${sid}:events`, JSON.stringify(newEvent));
await redis.ltrim(`session:${sid}:events`, -50, -1); // Keep last 50

Why 50 events?

  • Covers typical user journey (create → login → 10-20 attribute updates → logout)
  • Older events archived to cold storage (PostgreSQL) if needed
  • Keeps session size predictable (~5KB max)

Optimization 3: Separate Hot vs Cold Fields

Concept:

  • Hot fields: Updated frequently (lastSeenAt, state)
  • Cold fields: Rarely change (userId, createdAt, attributes)

Advanced Strategy: Separate Keys 

// Hot key: Updated every ping (short cache TTL)
HSET session:abc:hot lastSeenAt "2026-02-25T10:15:00Z"
HSET session:abc:hot pingCount 42
EXPIRE session:abc:hot 120  // 2 min cache

// Cold key: Rarely updated (longer cache TTL)
HSET session:abc:cold userId "alice"
HSET session:abc:cold state "authenticated"
HSET session:abc:cold createdAt "2026-02-25T10:00:00Z"
HSET session:abc:cold attributes '{"cart":["laptop"]}'
EXPIRE session:abc:cold 1800  // 30 min

// Activity ping only touches hot key (smaller write)
await redis.hset(`session:${sid}:hot`, 'lastSeenAt', now);
await redis.expire(`session:${sid}:hot`, 120);

Benefits:

  • Activity pings only write 24 bytes to hot key (not 5KB full session)
  • Cold data can be cached longer (CDN, application cache)
  • Hot data expires faster if user goes inactive (smaller Redis footprint)

Note: This optimization is only needed at very high scale (50k+ req/sec). For most applications, Redis Hashes alone provide sufficient performance.

Expiration Strategy

Key Design Decision: Sessions use sliding window TTL for security (short-lived), while devices use fixed TTL for convenience (long-lived "Remember Me").
Resource TTL Type Rationale
Session 30 minutes Sliding Window Security: Short-lived credentials prevent session hijacking
Device 90 days Fixed Convenience: "Remember Me" functionality for returning users
User Device Set None Permanent Audit: Track all devices ever used by user

Sliding Window Example

User Active Browsing:
  • 10:00 - Create session → Expires at 10:30
  • 10:15 - Click page (activity) → Expires at 10:45 ← Slides 30min
  • 10:40 - Update cart → Expires at 11:10 ← Slides again
  • 11:00 - Close browser → Expires at 11:30 ← No more activity

Result: Session stays alive while user is active, expires 30min after last action.

📌 Cookie Reading: Signed vs Unsigned

All cookies are signed for integrity. The server reads from: 

// ✅ Correct: Read signed cookies (signature verified)
const sessionId = req.signedCookies.sid;
const deviceId = req.signedCookies.did;

// ❌ Wrong: Would read unsigned cookies (no signature verification)
const sessionId = req.cookies.sid;

Why? Cookie signatures prevent attackers from forging session IDs. If an attacker modifies the cookie value (e.g., changing sid=abc to sid=xyz), the HMAC signature becomes invalid and req.signedCookies.sid returns undefined.

Attack Prevention

Session Fixation Attack ❌ PREVENTED

Attack Scenario:

  1. Attacker creates session: sid=attacker-123
  2. Attacker tricks victim to use this session
  3. Victim logs in with attacker's session
  4. Attacker now has access to victim's authenticated session

Our Defense:

Session Rotation on Login

  • When user logs in, old session ID (attacker-123) is deleted
  • New session ID (new-user-456) is created
  • Attacker's session becomes invalid, cannot access victim's account
XSS Cookie Theft ❌ PREVENTED

Attack Scenario:

// Malicious script tries to steal cookie
document.cookie; // Attacker wants to read sid/did

Our Defense:

HttpOnly Flag

  • Cookies set with httpOnly: true
  • JavaScript cannot access sid/did cookies
  • document.cookie returns empty (cookies invisible to JS)
CSRF Attack ⚠️ PARTIAL PROTECTION

Attack Scenario:

<!-- Malicious site: attacker.com -->
<!-- Scenario 1: Cross-site POST (BLOCKED) -->
<form action="https://yoursite.com/sessions/me/invalidate" method="POST">
  <input type="submit" value="Click to win!">
</form>

<!-- Scenario 2: Cross-site GET via link (COOKIES SENT) -->
<a href="https://yoursite.com/some-state-change?action=delete">
  Click here!
</a>

Current Defense: SameSite=Lax

All cookies use sameSite: "lax", which provides baseline CSRF protection:

  • ✅ Blocks cross-site POST: Forms submitted from attacker.com do NOT send cookies
  • ✅ Blocks cross-site fetch: fetch() requests from attacker.com do NOT send cookies
  • ⚠️ Allows top-level navigation: Links clicked from attacker.com DO send cookies (for UX reasons)

Why "lax" instead of "strict"?

  • sameSite: "strict" blocks cookies even on safe top-level navigation
  • Would break OAuth callback flows (user logged out after OAuth redirect)
  • Would break legitimate links from email/social media

🔐 Production CSRF Protection Strategy

Current protection: sameSite=lax handles most CSRF attacks, but state-changing endpoints need additional protection.

Recommended Implementation:

Option 1: CSRF Token (Recommended)
// 1. Issue token on session creation
POST /sessions
Response: {
  sessionId: "abc-123",
  csrfToken: "xyz-789",  // ← Store in session, return to client
  ...
}

// 2. Client stores token (localStorage or state)
localStorage.setItem('csrfToken', data.csrfToken);

// 3. Client sends token on state-changing requests
PATCH /sessions/me
Headers: {
  "X-CSRF-Token": "xyz-789"  // ← Custom header (can't be set by simple link)
}

// 4. Server validates token matches session
if (req.headers['x-csrf-token'] !== session.csrfToken) {
  return res.status(403).json({ error: "Invalid CSRF token" });
}

Why this works:

  • Attackers can't read the token (Same-Origin Policy prevents cross-site JavaScript access)
  • Simple links/forms can't set custom headers (only full JavaScript access)
  • Token rotation on identity attachment (like session ID rotation)
Option 2: Origin/Referer Validation
// Validate request comes from same origin
const origin = req.headers.origin || req.headers.referer;
const allowedOrigins = [process.env.APP_URL, 'https://yoursite.com'];

if (!origin || !allowedOrigins.some(o => origin.startsWith(o))) {
  return res.status(403).json({ error: "Invalid origin" });
}

Limitations:

  • Referer header can be missing (privacy extensions, HTTPS → HTTP)
  • Origin header not sent on same-site navigation (only cross-origin)
  • Less reliable than CSRF token approach
Affected Endpoints (Need CSRF Protection):
  • POST /sessions/me/activity (keep-alive ping)
  • PATCH /sessions/me (update attributes) ⚠️ High risk
  • POST /sessions/me/attach-identity (login) ⚠️ CRITICAL
  • POST /sessions/me/invalidate (logout) ⚠️ High risk
  • POST /sessions/merge (merge sessions) ⚠️ High risk

⚠️ Current implementation: Prototype/demo without CSRF tokens. Production deployment requires adding CSRF token validation to all state-changing endpoints.

Brute Force Session ID ❌ PREVENTED

Attack Scenario:

Attacker tries to guess valid session IDs

Our Defense:

UUID v4 (122 bits of randomness)

  • 2^122 = 5.3 × 10^36 possible session IDs
  • At 1000 guesses/second for 30 minutes = 1.8M attempts
  • Success rate: 1.8M / 5.3×10^36 ≈ 0% (effectively impossible)

🔒 CSRF Protection (CRITICAL)

  • Add CSRF token generation on POST /sessions
  • Store CSRF token in session data (Redis)

    • POST /sessions/me/activity

    • POST /sessions/me/invalidate
    • POST /sessions/merge

⚠️ Rate Limiting Implementation (CRITICAL)

Abuse Risk: Without rate limiting, endpoints like POST /sessions/me/activity can be spammed thousands of times per second, causing Redis write exhaustion and potential DoS.

POST /sessions HIGH 10 req/min per IP
POST /sessions/me/attach-identity HIGH 5 req/5min per IP (brute force)

Redis-Based Rate Limiting Implementation: 
// Middleware: Redis counters with TTL (atomic operations)
async function rateLimitMiddleware(scope, limit, windowSec) {
  return async (req, res, next) => {
    // Determine identifier (IP or session)
    const identifier = scope === 'ip' 
      ? req.ip 
      : req.signedCookies.sid;
    
    if (!identifier) {
      return next(); // Skip if no identifier
    }

    const key = `ratelimit:${scope}:${identifier}`;

    try {
      // Atomic increment
      const count = await redis.incr(key);
      
      // Set expiry on first request
      if (count === 1) {
        await redis.expire(key, windowSec);
      }

      // Check limit
      if (count > limit) {
        const ttl = await redis.ttl(key);
        return res.status(429).json({
          success: false,
          error: 'Rate limit exceeded',
          retryAfter: ttl,
          limit: limit,
          window: windowSec,
        });
      }

      // Pass through with rate limit headers
      res.setHeader('X-RateLimit-Limit', limit);
      res.setHeader('X-RateLimit-Remaining', Math.max(0, limit - count));
      res.setHeader('X-RateLimit-Reset', Date.now() + (ttl * 1000));
      
      next();
    } catch (error) {
      // Fail open: if Redis is down, allow request
      console.error('Rate limit check failed:', error);
      next();
    }
  };
}

// Apply to endpoints
app.post('/sessions/me/activity', 
  rateLimitMiddleware('session', 60, 60), // 60 req/min per session
  async (req, res) => { /* ... */ }
);

app.post('/sessions', 
  rateLimitMiddleware('ip', 10, 60), // 10 req/min per IP
  async (req, res) => { /* ... */ }
);

app.post('/sessions/me/attach-identity', 
  rateLimitMiddleware('ip', 5, 300), // 5 req/5min per IP
  async (req, res) => { /* ... */ }
);

Why Redis for Rate Limiting?
  • Atomic operations: INCR + EXPIRE are atomic, no race conditions
  • Automatic cleanup: TTL-based keys expire automatically
  • Shared state: Works across multiple server instances
  • Fast: In-memory lookups, <1ms overhead
Trade-offs: Fixed vs Sliding Window

Fixed Window (simpler, used above):

  • ✅ Simple implementation (INCR + EXPIRE)
  • ❌ Burst at window boundaries (60 req at 00:59, 60 at 01:00 = 120 in 2 sec)
  • Use case: Good enough for most APIs

Sliding Window (more accurate):

  • ✅ Smooth rate limiting (true X req/min at any moment)
  • ❌ More complex (ZSET with timestamps + ZREMRANGEBYSCORE)
  • Use case: High-value endpoints (attach-identity)

⚠️ Current prototype: NO rate limiting implemented. Production deployment MUST add rate limiting to prevent abuse, especially on POST /sessions/me/activity.

  }

});

⚠️ Common Pitfalls:

  • ❌ Using Access-Control-Allow-Origin: * with credentials (not allowed by spec)

  • Same-origin: Remove or disable CORS middleware (slight perf gain)

Read vs Write Path Analysis

Traffic Pattern: 10:1 Read/Write Ratio

Read Operations (90%):

  • GET /sessions/me (session fetch)
  • POST /sessions/me/activity (TTL refresh)
  • GET /internal/sessions/:sid (cross-service)

Write Operations (10%):

  • POST /sessions (create)
  • PATCH /sessions/me (update attributes)
  • POST /sessions/me/attach-identity (login)

Optimization: Read Replicas

Route 90% of traffic to read replicas, reserve master for writes. Redis replication lag: <10ms (acceptable for sessions).

⚠️ Rate Limiting: Critical for Capacity Protection

Abuse Scenario Without Rate Limiting:

Attack: Single malicious client spams POST /sessions/me/activity

  • Volume: 10,000 requests/second (easily achievable from a single machine)
  • Redis impact: 10k writes/sec = 20% of single Redis capacity
  • Result: 5 attackers can saturate Redis master, degrading service for all users
  • Cost: If auto-scaling, could spin up unnecessary infrastructure

Protection: Per-Session Rate Limiting

Limit: 60 requests/minute per session (1 req/sec) on activity endpoint

  • Legitimate use: Frontend pings every 60 seconds = 1 req/min (well under limit)
  • Aggressive polling: Every 10 seconds = 6 req/min (still allowed)
  • Attack blocked: 1000 req/min = 940 requests return 429 (no Redis writes)
  • Result: Even with 1M active sessions, max Redis write load = 1M req/min = 16k req/sec (within capacity)

💡 Rate limiting converts unbounded abuse into predictable, manageable load.

Expected Bottlenecks

1. Redis Memory At 1M sessions: 5GB RAM used. Redis OOM = service down.
Solution: Monitoring + alerts at 80% capacity, Redis eviction policy (allkeys-lru).
2. Network Bandwidth 10k ops/sec × 5KB = 50MB/sec. Saturates 1Gbps link at 20k ops/sec.
Solution: 10Gbps NICs, compression for large attributes.
3. Single Point of Failure One Redis instance = no redundancy.
Solution: Redis Sentinel (3-node HA setup) with automatic failover.
4. Cross-Service Latency Internal API adds network hop (~5ms).
Solution: Service-local cache (L1) for frequently accessed sessions.

Caching Strategy

L1 Cache (Application Memory):

Each service maintains a local LRU cache (1000 sessions, 5MB) for frequently accessed sessions. Cache hit rate: ~30-40% for internal API calls. 

// Service-side cache
const sessionCache = new LRU({ max: 1000, ttl: 60000 }); // 1min TTL

const session = sessionCache.get(sid) || await fetchFromSessionService(sid);
sessionCache.set(sid, session);

Tradeoff: Stale data risk (60sec), but 5ms latency savings on cache hits.

Tradeoffs & Alternatives

Key Design Decisions

Why Redis over JWT?
Factor Redis (Chosen) JWT (Alternative)
Revocation Instant - Delete key Impossible - Token valid until expiry
Multi-Device Native - Redis Sets Complex - Needs database anyway
Mutable Sessions Easy - Update Redis value Hard - Re-issue token on every change
State Size Unlimited - Server-side Limited - Cookie size limits (~4KB)
Latency ~5ms - Network + Redis ~0ms - No lookup needed
Scalability Stateful - Needs Redis cluster Stateless - No shared state

Verdict: Redis chosen because revocation and multi-device tracking are hard requirements. JWT better for stateless microservices without these needs.

Why Not PostgreSQL for Sessions?

Performance Gap:

  • PostgreSQL indexed query: ~10-50ms (disk I/O bottleneck)
  • Redis in-memory lookup: ~1-3ms (no disk)
  • At 10k req/sec: PostgreSQL saturates, Redis has 5× headroom

When PostgreSQL Makes Sense:

  • Cold storage (archived sessions for audit)
  • Complex queries ("find all sessions from IP X")
  • ACID transactions across sessions + orders + payments
Why Cookies over Bearer Tokens?

Security Benefits:

  • HttpOnly: JavaScript cannot steal cookie (XSS protection)
  • SameSite=Lax: Baseline CSRF protection (blocks cross-site POST)
  • Browser-managed: Automatic expiry, no localStorage risks

Bearer Token Use Case:

Mobile apps, API-to-API auth where cookies don't work. For those, use Authorization: Bearer <token> with device ID.

Alternative Architectures Considered

Option A: Fully Stateless (JWT + Database)

Architecture: JWT in cookie, session metadata in PostgreSQL

Pros:

  • Infinite horizontal scaling (stateless servers)
  • No Redis dependency
  • Lower operational complexity

Cons:

  • ❌ Cannot revoke sessions instantly (JWT valid until expiry)
  • ❌ Logout requires blacklist (back to stateful anyway)
  • ❌ Multi-device tracking requires database queries
  • ❌ Cookie size limits (4KB) restrict session data

Verdict: Rejected due to revocation requirement.

Option B: Database-Only Sessions (PostgreSQL)

Architecture: Sessions in PostgreSQL with indexed lookups

Pros:

  • ACID transactions (session + payment)
  • Rich queries (analytics, audit)
  • Persistent by default

Cons:

  • ❌ Latency: 10-50ms (misses <10ms target)
  • ❌ Throughput: 5k ops/sec (misses 10k target)
  • ❌ TTL requires background job (not native)

Verdict: Rejected due to performance gap. Good for cold storage.

Option C: Hybrid (Redis + PostgreSQL)

Architecture: Hot sessions in Redis, cold archive in PostgreSQL

Pros:

  • ✅ Low latency (<10ms) for active sessions
  • ✅ Historical data for analytics
  • ✅ Best of both worlds

Cons:

  • Higher operational complexity (2 datastores)
  • Sync jobs required (Redis → PostgreSQL hourly)

Verdict: ⭐ Recommended for production at scale. Current implementation (Redis-only) is Phase 1.
